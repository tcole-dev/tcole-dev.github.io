事务与MVCC

事务特性

ACID特性

  • Atomicity - 原子性：

    和JUC中的原子性差不多，其内部的修改，要么全部成功，要么全部失败。

    若事务中止，其执行过的修改操作，通过回滚，全部不可见；若事务提交，其执行的修改才能被看见。但本质上事务并不能真正成为原子操作，其中间态在读未提交隔离级别中仍然可见。

  • Consistency - 一致性：

    事务执行前后，数据库都必须满足预先定义好的规则和约束。简单地来说就是事务是从一个合法状态到另一个合法状态，不能停留在一个违反规则（数据库的约束、业务不变量如账户余额…等）的状态。

    一致性是事务的属性，更是事务的目的，其他特性都是为一致性服务的。

  • Isolation - 隔离性：

    隔离性是指，不同的事务之间不能互相干扰，一个事务内部的操作和数据，不能被其他事务获取，反之亦然。隔离性越高，越能保证数据的一致性，但对性能影响越大，由此会引出四种隔离级别。

  • Durability - 持久性：

    持久性指事务提交后，数据变更就是永久的，即使出现数据库崩溃等问题，也可以通过日志等找回。

一致性问题

读一致性问题：脏读、不可重复读、幻读问题

  • 脏读：

    事务执行中可读取到其他事务还未提交的数据，原本数据为A、一个事务修改后为B，但这个事务有中间态会修改数据为C，那么另一个事务可能就会读取到C，读取到其他事务的中间态，就是脏读。（即使这个中间态后续不会再修改，会被直接提交）

  • 不可重复读：

    事务执行中，第一次读取的数据为A，但第二次读取时，这条记录的数据被修改，变成了B，同一个事务中，读取同一条记录，但数据不同，这就是不可重复读。

  • 幻读：

    事务执行中，第一次查询时，只有三条记录，但另一个事务因为update、insert、delete使得第二次查询时，原本的三条记录变成四条、两条或其他值。某些数据（记录）时有时无，就是幻读。

    不可重复读是一条数据的不一致，幻读是数据数量的不一致。

除读一致性外，还有写一致性：丢失更新、写写冲突、脏写，写一致性一般由锁机制解决，在后面的学习中再记录。

事务隔离级别

事务隔离级别有四个：读未提交、读已提交、可重复读、串行化。事务隔离是解决读一致性问题的一大实现。

以上面的例子来说，四个隔离级别：

  • 读未提交：

    事务可以读取到其他事务未提交的数据（获取到中间态），在上面的例子中，V1、V2、V3的返回值分别是 2、2、2，三个读一致性问题，这种隔离级别下，一个都解决不了。

  • 读已提交：

    事务只能读取到其他事务提交的数据，可以解决脏读，但无法解决不可重复读、幻读。V1、V2、V3返回值分别是 1、2、2

  • 可重复读：

    在同一个事务内部，读取某一条数据的值总是一样的，这种隔离级别可以解决脏读、不可重复读，但无法解决幻读的问题。V1、V2、V3的返回值为 1、1、2

  • 串行化：

    串行化对于同一行记录，读会加锁（共享锁）、写也会加锁（互斥锁），当锁冲突时，就会阻塞后一个事务，让前面的事务执行完毕，就像是让同时执行的事务按某种串行顺序一个一个执行。V1、V2、V3的返回值为 2、2、2（虽然和读未提交结果相同，但两者的含义完全不同）。

串行化最安全，但对性能影响最大，MySQL默认使用的是可重复读，Oracle默认使用的是读已提交。

事务开启方式

  • BEGIN / START TRANSACTION：两者都是显式开启事务，同时搭配COMMIT / ROLL BACK
  • set autocommit=0：执行一条SQL语句时，默认开启事务，但不会关闭，直到手动commit，同时commit后的下一条SQL又会开启事务
  • set autocommit=1：任意一条SQL语句都是一次事务，执行完后自动提交

MVCC机制

什么是MVCC

在早期的数据库系统中，通常使用锁来实现事务隔离，而这种方式通常会导致大量冲突，除读读外，读写、写读、写写都会导致阻塞，如串行化中，读写同时操作就会导致阻塞。这样的方式虽然安全，但会降低并发性能。

MVCC，多版本控制，和锁一样用于实现事务隔离，但通过保留数据的多个版本，使得读写不再冲突阻塞，提高并发性能，但写写操作时，还是需要加锁保证一致性。

MVCC实现原理

版本链

相关概念：

undo log：undo log是专门用于事务回滚、MVCC的日志，用于存放不同事务修改后的数据。

隐藏字段：对于InnoDB存储引擎，每一行记录都有两个隐藏列trx_id、roll_pointer,如果数据表中存在主键或者非NULL的UNIQUE键时不会创建row_id,否则InnoDB会自动生成单调递增的隐藏主键row_id。

列名 是否必须 描述
row_id 否 单调递增的行ID，不是必需的，占用6个字节。 这个跟MVCC关系不大
trx_id 是 记录操作该行数据事务的事务ID
roll_pointer 是 回滚指针，指向当前记录行的undo log信息

多个事务并行操作某一行数据时，不同事务对该行数据的修改会产生多个版本，然后通过回滚指针（roll_pointer），连成一个链表，这个链表就称为版本链。

ReadView

Read View就是一个快照，保存着数据库某个时刻的数据信息。Read View会根据事务的隔离级别决定在某个事务开始时，该事务能看到什么信息。就是说通过Read View，事务可以知道此时此刻能看到哪个版本的数据记录（有可能不是最新版本的，也有可能是最新版本的）。可重复读、读已提交，这两个隔离级别的普通 SELECT 都会使用Read View。

ReadView用于快照读，所以串行化不使用ReadView，因为它是当前读。对于读未提交也不适用，因为读未提交可以脏读。

ReadView的原理:

  • trx_ids: 当前系统中那些活跃(未提交)的读写事务ID, 它数据结构为一个List。(重点注意:这里的trx_ids中的活跃事务，不包括当前事务自己和已提交的事务，这点非常重要)
  • low_limit_id: 目前出现过的最大的事务ID+1，即下一个将被分配的事务ID。
  • up_limit_id: 活跃事务列表trx_ids中最小的事务ID，如果trx_ids为空，则up_limit_id 为 low_limit_id。
  • creator_trx_id: 表示生成该 ReadView 的事务的 事务id

判断某条记录的某个版本是否对当前事务可见，判断规则如下（注意：ReadView是某一时刻的快照）：

  • 如果被访问版本的 事务ID = creator_trx_id，那么表示当前事务访问的是自己修改过的记录，那么该版本对当前事务可见；

  • 如果被访问版本的 事务ID < up_limit_id，那么表示生成该版本的事务在当前事务生成 ReadView 前已经提交，所以该版本可以被当前事务访问。

  • 如果被访问版本的 事务ID > low_limit_id 值，那么表示生成该版本的事务在当前事务生成 ReadView 后才开启，所以该版本不可以被当前事务访问。

  • 如果被访问版本的 事务ID在 up_limit_id和m_low_limit_id 之间，那就需要判断一下版本的事务ID是不是在 trx_ids 列表中。

    如果在，说明创建 ReadView 时生成该版本的事务还是活跃的，该版本不可以被访问；
    如果不在，说明创建 ReadView 时生成该版本的事务已经被提交，该版本可以被访问。

当前读 & 快照读

前面提到，MVCC用于快照读，判断的是不同快照之间的时间关系，由此判断是否可见。

而当前读一定要获取到最新值，所以当前读需要显式加锁。

不同隔离级别的不同表现

MVCC主要用于RR、RC两种隔离级别的 快照读，在读未提交、串行化中均不起主要作用。而在 RC 、 RR 中，MVCC本身的行为区别不大，核心机制都是版本链、生成ReadView，主要核心区别就在于ReadView的生成、复用时机不同。

前面提到ReadView本质上是某一时刻的快照，那么比如RR中是如何读取最新提交的？

原因在于： 在RR中，ReadView在第一次快照读时生成，后续读取都复用这个ReadView快照；

​ 在RC中，每一次快照读都会生成新的ReadView。

总之，不同隔离级别中，MVCC核心机制相同，ReadView判断规则也相同（如上），但ReadView的生成、复用的时机不同造成了差异。

总结

在这一篇中，学习、记录了数据库操作中的事务，包括事务的四大特性：原子性、可见性、隔离性、持久性，其中可见性是事务的核心，其他特性都是为可见性服务的。

由此，引出了可见性问题（读可见性、写可见性，这里主要是读可见性）：脏读（读到未提交数据）、不可重复读（多次读取不一致）、幻读（时有时无），为了避免可见性问题，有了事务隔离级别。

继而又引出了事务隔离级别如何实现：早期的完全靠锁 -> MVCC + 锁。MVCC的实现机制是 版本链存储不同版本信息、ReadView可见性判断。不同隔离级别下，主要是ReadView的生成、复用时机不同实现不同效果。

事务 -> 可见性问题 -> 事务隔离 -> 隔离级别的实现 -> MVCC -> MVCC的实现。

MVCC用于快照读，解决的是读读、读写、写读的冲突，而还有许多情况需要通过**锁来实现，比如写写之间的冲突（写可见性）、当前读（需要显式加锁），数据库的锁**就是后面需要深入的部分。
