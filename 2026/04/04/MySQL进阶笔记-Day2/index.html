索引结构

聚簇索引和非聚簇索引

聚簇索引：索引和数据是存放在一起的，同时表中数据会按照索引顺序排列存储，索引即数据，两者严格绑定。InnoDB引擎中，主键为聚簇索引，且一张表只有一个聚簇索引，因为表中记录是按照聚簇索引排列的。

非聚簇索引：InnoDB引擎中，二级索引，如INDEX、NOT NULL等都是非聚簇索引，非聚簇索引同样会产生一颗B+树，但这颗B+树的叶子节点中，不存储完整记录，而是存储主键或数据地址，当通过非聚簇索引查找时，通常需要通过非聚簇索引查找到该记录的主键，再根据主键（聚簇索引）查找完整记录。

id作为主键，是聚簇索引，因此底层的B+树中，是按照id的值排列的，表的顺序就是索引顺序。根据id可以查询到完整记录。

而非聚簇索引中叶子节点只存索引（key）和主键（value），再根据主键再次查找，这种做法就叫回表查询

正因为回表查询的存在，所以实际使用中，应该尽量使用主键（聚簇索引），而非二级索引进行条件筛选。

不同存储引擎的索引

前面提到在InnoDB引擎中，表数据文件本身就是按 B+Tree 组织的一个索引结构，索引就是数据，二级索引（非聚簇索引）是另外单独的一个B+树。

而在MyISAM中，完全使用非聚簇索引，数据与索引完全分离，MyISAM 是“B+树索引”指向“堆文件”，本身的数据并不是B+树，数据文件 (.MYD) 的结构是一块连续的磁盘空间，数据按照insert的顺序一行一行地排列，而索引 (.MYI) 则是B+树，B+树中记录索引键和行记录的物理地址。

索引下推（ICP）

索引下推和覆盖索引有些异曲同工之妙，它们都是为了减少回表而产生的机制，其中，覆盖索引是解决是否需要回表，而索引下推则是尽可能减少回表次数。

前面提到，若查询语句查询的字段恰好都被联合索引覆盖，则不需要回表，直接在联合索引这张“小表”中查询返回即可。而另一种情况：若查询的字段没有被完全覆盖，则需要进行回表，这就产生了一个问题，举个例子来说：

索引失效

在很多场景下，索引可能会失效，或者优化器觉得索引收益过低，这时就会直接进行全表扫描，常见索引失效场景如下：

较常见情况：

场景 典型 SQL 失效类型 原因 推荐做法
对索引列使用函数 where date(create_time) = '2026-04-06' 完全失效 函数作用在列上，无法直接利用原始索引值 改为范围查询：create_time >= ... and create_time < ...
对索引列做运算/表达式 where age + 1 = 20 完全失效 列值先参与计算，索引无法直接定位 改写为 where age = 19
like 以 % 开头 where name like '%abc' / '%abc%' 完全失效 B+Tree 索引不能从左侧确定起始范围 尽量改成 like 'abc%'
联合索引不满足最左前缀 索引 (a,b,c)，查询 where b = 1 完全失效 跳过最左列，无法从索引头开始匹配 查询尽量从最左列开始，或重建索引
联合索引中范围查询后续列失效 索引 (a,b,c)，查询 where a=1 and b>2 and c=3 部分失效 b 出现范围后，c 往往不能继续用于定位 等值条件尽量放前，按查询模式设计索引
隐式类型转换 where phone = 13800138000，而 phone 是 varchar 完全/大概率失效 比较时可能发生列类型转换，导致索引不能正常利用 保持字段类型和参数类型一致
or 使用不当 where a=1 or b=2 容易失效 多条件合并成本高，优化器可能放弃索引走全表 可改成 union all 分开查
使用 !=、<>、not in where status != 1 容易失效 否定条件选择性差，扫描范围通常很大 尽量改为正向条件
低区分度字段建索引 where is_deleted = 0 容易不走索引 重复值太多，走索引后仍需扫描大量数据 低区分度字段不宜单独建索引，可做联合索引
查询结果集过大 select * from user where age > 18 容易不走索引 命中记录太多，回表成本高，优化器可能选全表扫描 缩小范围、避免大结果集
select * 导致大量回表 select * from t where a = 1 索引利用差 条件能走索引，但查询列太多导致频繁回表 尽量只查需要的列，使用覆盖索引
排序与索引顺序不一致 索引 (a,b)，order by b,a 排序阶段失效 索引顺序无法满足排序要求，可能触发 filesort 让 where 与 order by 顺序匹配索引

非常见情况：

场景 典型 SQL / 情况 失效类型 原因 推荐做法
统计信息过旧 有索引但 EXPLAIN 不走 伪失效 不是索引真失效，而是优化器因统计信息不准选错执行计划 更新统计信息，重新分析执行计划
字符集 / 排序规则不一致 两个字符串列比较或参数字符集不同 容易失效 比较时可能发生转换，影响索引使用 统一字符集与排序规则
联合索引“看似命中，实则部分失效” 索引 (a,b,c)，查询 where a=1 and c=3 部分失效 只能稳定利用 a，c 通常只参与过滤 按真实查询模式补建索引 (a,c)
范围条件叠加排序 索引 (a,b)，查询 where a > 10 order by b 排序部分失效 过滤后无法继续保证 b 的全局有序性 重新设计过滤与排序字段顺序
group by 无法匹配索引 索引 (a,b)，group by b 分组阶段失效 分组字段不满足索引前缀要求，可能使用临时表 让分组字段顺序匹配索引
深分页 limit 100000, 20 非严格失效，但效果很差 即便走索引，也要扫描前面大量记录再丢弃 用游标分页、延迟关联
分区裁剪未命中 分区表查询未带分区键 分区级退化 无法缩小到目标分区，导致扫描多个分区 查询条件尽量包含分区字段
表达式索引未正确命中 建了表达式索引，但 SQL 表达式写法不同 失效 查询表达式与索引定义不一致 查询表达式与索引定义保持一致
生成列索引未按生成逻辑使用 对生成列建索引，但查询写原始复杂表达式 容易失效 优化器不一定自动等价改写 直接查询生成列
参数绑定类型错误 代码里预编译参数类型传错 隐蔽失效 线上 SQL 看起来没问题，但驱动传参类型不对 检查 ORM / 驱动参数类型
复合查询优化器改写导致未按预期走索引 子查询、视图、复杂 OR/AND 嵌套 间接失效 SQL 被改写后执行计划变化，原索引未被优先使用 拆解复杂 SQL，逐段 EXPLAIN
覆盖索引缺失导致优化器放弃索引 条件字段有索引，但回表列太多 间接失效 优化器评估回表成本高，直接放弃索引 建覆盖索引或减少查询列
